### Автоматизация подбора вакансий на HH.ru

> RSS (Really Simple Syndication — «действительно простое получение информации») — это технология, которая позволяет автоматически получать обновления с веб-сайтов, блогов, новостных порталов и других ресурсов с регулярными публикациями. Она собирает новые материалы в одном месте, без необходимости вручную заходить на каждый сайт

С декабря 2025 года напрямую получить доступ к API `hh.ru` не получится (для соискателей его упразднили). При этом доступ к RSS ресурса сохраняется

Поэтому идея следующая:
- Получить список новых вакансий по запросу с заданными фильтрами (по региону, требуемому опыту и т.д.) из ленты публикации вакансий
- Из полученных данных вытянуть ссылки на карточки вакансий
- Разобрать HTML-страницы вакансий, вытянуть из них необходимые данные
- Отправить получившуюся выжимку по вакансии в Telegram-чат

#### Получение вакансий

Запросы направляются на
```
https://hh.ru/search/vacancy/rss
```

В запросе можно указать фильтры (параметры). Вот некоторые из них
- `text` — произвольный поисковой запрос (например, `DevOps`, `Системный администратор`, `Специалист технической поддержки L2` и т.п.)
- `work_format`
  - В офисе: `ON_SITE`
  - Удаленно: `REMOTE`
  - Гибрид: `HYBRID`
- `employment_form` — тип занятости
  - Полная занятость: `FULL`
- `area` — регион поиска
  - Вся Россия: `113`
  - Новосибирская область `1202`
  - ...
- `experience` — опыт работы
  - Если не указать, то аналогично параметру "Не важно"
  - Без опыта: `noExperience`
  - От года до 3х лет: `between1And3`
  - От 3х до 6 лет: `between3And6`

Пример URL запроса с параметрами
```
"https://hh.ru/search/vacancy/rss?text=DevOps&area=113&employment_form=FULL&experience=between1And3"
```

#### Теги HTML, которые можно использовать в сообщении Telegram

Базовое форматирование:
- `<b>...</b>` или `<strong>...</strong>` — жирный текст
- `<i>...</i>` или `<em>...</em>` — курсив
- `<u>...</u>` — подчеркнутый текст
- `<s>...</s>` или `<strike>...</strike>` или `<del>...</del>` — зачеркнутый
- `<tg-spoiler>...</tg-spoiler>` — спойлер (скрытый текст, появляется по нажатию)

Моноширинный текст:
- `<code>...</code>` — моноширинный (inline-блок)
- `<pre>...</pre>` — блок кода (сохраняет форматирование)
- `<pre><code class="language-python">...</code></pre>` — блок кода с подсветкой (укажите язык)

Ссылки и упоминания:
- `<a href="URL">...</a>` — текстовая гиперссылка
- `<a href="tg://user?id=123456789">...</a>` — ссылка на профиль пользователя

Цитаты:
- `<blockquote>...</blockquote>` — блок цитирования
- `<blockquote expandable>...</blockquote>` — сворачиваемая цитата

### Интерфейс командной строки для управления приложением

Утилита `subjob` предоставляет 4 команды для управления приложением из терминала. Базовый синтаксис:

```
subjob [КОМАНДА]
```

#### Инициализация базы данных `subjob init`

Инициализирует базу данных SQLite. Создаёт каталог `data/` (если не существует) и файл `data/rss_subscriptions.db` с таблицами:

| Таблица | Назначение |
|---|---|
| `rss_subscriptions` | Хранит подписки пользователей: `id`, `user_id`, `rss_url` |
| `sent_vacancies` | Учёт уже отправленных вакансий для предотвращения отправки дубликатов: `user_id`, `vacancy_url`, `sent_at` |

Индексы: 
- `rss_subscriptions_user_id` — ускоряет поиск подписок по пользователю.

#### Запуск Telegram-бота `subjob runbot`

Запускает Telegram-бота в режиме длительного опроса обновлений (long polling). Бот бесконечно ожидает новые сообщения от пользователей и выполняет фоновую задачу проверки RSS-лент.


#### Отображение таблицы RSS-подписок `subjob sublist`

Выводит все RSS-подписки из базы данных в постраничной таблице. Колонки: `record_id`, `user_id`, `rss_link`. Пагинация — по 8 записей на странице. Навигация: `Enter` — следующая страница, `q` — выход.


#### Рассылка вакансий по подписке пользователям `subjob bcast`

Ручной запуск широковещательной рассылки. Парсит все RSS-ленты всех пользователей и отправляет новые вакансии тем, кто на них подписан. Полезно для отладки или внеочередной проверки обновлений.

#### Справка по командам

```
subjob --help
```

### Функционал Telegram-бота для взаимодействия с пользователем

Бот предоставляет 6 команд, из которых две (`/subscribe` и `/unsubscribe`) реализованы через **ConversationHandler** — класс из библиотеки `python-telegram-bot`, который управляет многошаговыми диалогами.

**ConversationHandler** связывает последовательность шагов в единый сценарий, храня промежуточное состояние в пользовательском контексте (`context.user_data`).

Основные концепции
- **entry_points** — команды, запускающие диалог (например, `/subscribe`)
- **states** — словарь, где ключом выступает имя состояния (константа), а значением — список обработчиков, активных *только* в этом состоянии
- Каждый обработчик возвращает **следующее состояние** (или `ConversationHandler.END` для завершения диалога)
- **fallbacks** — команды типа `/cancel`, которые прерывают диалог из любого состояния
- Все промежуточные данные живут в `context.user_data` и доступны на всех шагах диалога

```python
ConversationHandler(
    entry_points=[...],   # точка(и) входа — команды, с которых начинается диалог
    states={              # словарь состояний: на каждом шаге свой набор обработчиков
        STATE_1: [MessageHandler(...)],
        STATE_2: [CallbackQueryHandler(...)],
        ...
    },
    fallbacks=[...],      # команды, которые прерывают диалог в любой момент (например, /cancel)
)
```

#### Команды бота

##### Приветствие пользователя `/start`

Отправляет приветственное сообщение и предлагает ознакомиться со списком команд через `/help`.

##### Справочная информация по командам `/help`

Выводит список всех доступных команд с кратким описанием

##### Отображение активных подписок `/show_subscriptions`

Отображает все активные подписки текущего пользователя. Для каждой подписки парсит RSS-URL через `parse_rss_url_to_dict()`, преобразует параметры в читаемый вид (с эмодзи) и рендерит через Jinja2-шаблон `templates/rss_params_message.html`.

##### Оформление подписки `/subscribe` (ConversationHandler)

Пошаговый диалог из 5 состояний. На каждом шаге пользователю предлагаются inline-кнопки с эмодзи и кнопка «⏭ Пропустить». Отмена диалога (оформления подписки) — `/cancel` на любом шаге.

Последовательная схема работы:
```
Пользователь: /subscribe
      ↓
    subscribe_command_handler()  → запрашивает текст, возвращает SEARCH_TEXT
      ↓
Пользователь: "Python разработчик"
      ↓
    search_text_received()       → сохраняет текст, показывает кнопки регионов, возвращает REGION
      ↓
Пользователь: нажимает кнопку "Новосибирск"
      ↓
    region_selected()            → сохраняет регион, показывает кнопки для выбора формата работы, возвращает WORK_FORMAT
      ↓
Пользователь: нажимает "Удалённо"
      ↓
    work_format_selected()       → сохраняет формат, показывает кнопки для выбора типа занятости, возвращает EMPLOYMENT_FORM
      ↓
Пользователь: нажимает "Полная занятость"
      ↓
    employment_selected()        → сохраняет занятость, показывает кнопки для выбора опыта работы, возвращает EXPERIENCE
      ↓
Пользователь: нажимает "1-3 года"
      ↓
    experience_selected()        → вызывает create_rss_request_url(), выводит итог, сохраняет подписку в БД, возвращает END
```

Доступные значения фильтров (константы из `modules/constants/constants.py`):

| Фильтр | Ключ | Значения |
|---|---|---|
| Регион | `REGION_MAP` | `1202` → Новосибирская область, `113` → Россия (все регионы) |
| Формат работы | `WORK_FORMAT_MAP` | `ON_SITE` → 🏢 В офисе, `REMOTE` → 🏠 Удалённо, `HYBRID` → 🔄 Гибрид |
| Тип занятости | `EMPLOYMENT_MAP` | `FULL` → 📋 Полная занятость, `PART` → ⏳ Частичная занятость |
| Опыт работы | `EXPERIENCE_MAP` | `noExperience` → 🟢 Без опыта, `between1And3` → 🟡 1-3 года, `between3And6` → 🟠 3-6 лет, `moreThan6` → 🔴 Более 6 лет |

Финальный шаг:
1. Формирует RSS-URL с заданными фильтрами через `create_rss_request_url()`
2. Сохраняет подписку в БД через `add_rss_subscription()` (с проверкой на дубликат)
3. Выводит итоговую сводку выбранных параметров и ссылку на RSS-ленту

##### Удаление подписки`/unsubscribe` (ConversationHandler)

Двухшаговый диалог:
1. Показывает нумерованный список подписок пользователя, просит ввести номер для удаления
2. По введённому номеру удаляет подписку из БД через `delete_rss_subscription()` (с `RETURNING rss_url` для подтверждения)

##### Удаление всех подписок `/reset`

Удаляет все подписки текущего пользователя разом. Для каждой подписки вызывается `delete_rss_subscription()`. Выводит количество удалённых записей.

#### Фоновая задача: периодическая проверка RSS-лент

Функция `send_vacancies_info()` запускается автоматически через `app.job_queue.run_repeating`:

- **Интервал:** настраивается через `RSS_CHECK_INTERVAL_SECONDS` (по умолчанию 3600 сек = 1 час)
- **Первый запуск:** через 10 секунд после старта бота
- **Ограничение парсинга:** не более `VACANCIES_PARSE_LIMIT` вакансий с каждой RSS-ленты за один цикл проверки обновлений
- **Пауза между запросами:** 1 секунда между парсингом страниц вакансий (чтобы не нагружать HH.ru)

Алгоритм работы:
1. Получает все подписки, сгруппированные по пользователям (`dict_rss_subscriptions()`)
2. Группирует пользователей по уникальным RSS-URL — если у нескольких пользователей одинаковый URL запроса, лента парсится **один раз**
3. Для каждого уникального URL:
   - Загружает RSS-ленту (`parse_rss_feed()`)
   - Для каждой записи (в пределах лимита) парсит HTML-страницу вакансии (`parse_vacancy()`)
   - Рендерит карточку вакансии через Jinja2-шаблон `templates/jobs.html`
4. Рассылает результаты всем подписчикам URL
5. Дедупликация: перед отправкой проверяет, отправлялась ли уже пользователю такая вакансия (`is_vacancy_already_sent()`), после успешной отправки вызывает `mark_vacancy_as_sent()`

Извлекаемые данные из страницы вакансии (`parse_vacancy()`):

| Поле | data-qa атрибут | Значение при отсутствии |
|---|---|---|
| `title` | `vacancy-title` | "Не найдено" |
| `salary` | (класс `compensation-row`) | "Не указана" |
| `company` | `vacancy-company-name` | "Не указана" |
| `experience` | `work-experience-text` | "Не указан" |
| `schedule` | `work-schedule-by-days-text` | "Не указан" |
| `employment` | `common-employment-text` | "Не указано" |
| `working_hours` | `working-hours-text` | "Не указано" |
| `work_place` | `vacancy-view-raw-address` | "Не указан" |
| `work_format` | `work-formats-text` | "Не указан" |
| `description` | `vacancy-description` | "" |
| `url` | (URL страницы) | — |

Ключевые преимущества
- **Изоляция** — нажатие кнопки на шаге 3 не попадёт случайно в обработчик шага 2
- **Контекст** — все собранные параметры живут в `context.user_data` на протяжении всего диалога
- **Отмена** — `/cancel` в любой момент вызывает `subscribe_cancel()` или `unsubscribe_cancel()` и сбрасывает диалог
- **Дедупликация RSS** — одинаковые ленты парсятся один раз для всех подписчиков
- **Дедупликация вакансий** — отправленная вакансия не будет отправлена повторно

### Установка на чистую ВМ

#### Системные требования

- **Python** ≥ 3.12
- **uv** (рекомендуется) или **pip** ≥ 23.0
- **git**
- Доступ к Telegram API (токен бота от @BotFather)

#### Переменные окружения (файл `.env`)

| Переменная | Обязательность | Описание                                                                                                     | Значение по умолчанию |
|---|---|--------------------------------------------------------------------------------------------------------------|--------------|
| `TELEGRAM_API_TOKEN` | Обязательна | Токен Telegram бота, полученный от @BotFather                                                                | —            |
| `RSS_CHECK_INTERVAL_SECONDS` | Опциональна | Интервал автоматической проверки RSS-лент в секундах                                                         | `3600` (1 час) |
| `VACANCIES_PARSE_LIMIT` | Опциональна | Ограничение на количество вакансий, которые парсятся с каждой RSS-ленты за один цикл проверки новых вакансий | `3`          |

Пример `.env`:
```
TELEGRAM_API_TOKEN=1234567890:ABCdefGHIjklMNOpqrsTUVwxyz
RSS_CHECK_INTERVAL_SECONDS=1800
VACANCIES_PARSE_LIMIT=5
```

- `jinja2` — рендеринг HTML-шаблонов сообщений

#### Запуск в фоновом режиме (Linux)

> Далее предполагается, что пользователь клонировал проект в каталог `/opt`. То есть рабочая директория проекта это `/opt/subscribe_job_rss`. Если проект был клонирован в другое место, необходимо скорректировать соответствующие пути в конфигурации сервиса

Для долговременной работы бота рекомендуется использовать systemd-сервис:

```ini
# /etc/systemd/system/subjob.service
[Unit]
# Человекочитаемое имя модуля systemd
Description=SubJob Telegram Bot
# Запуск только после того, как сеть будет поднята (т.к. боту нужен интернет)
After=network.target

[Service]
# systemd выполняет команду запуска, которая создает один процесс (экземпляр приложения)
Type=simple
# Запускать службу от имени пользователя subjob (а не от root)
User=subjob
# Рабочая директория, относительно которой вычисляются пути (каталог, в который клонирован проект)
WorkingDirectory=/opt/subscribe_job_rss
# Для того, чтобы systemd нашел исполняемый файл subjob, нужно указать путь, где он располагается после установки
Environment="PATH=/opt/subscribe_job_rss/.venv/bin:/usr/local/bin:/usr/bin"
# Команда запуска приложения
ExecStart=/opt/subscribe_job_rss/.venv/bin/subjob runbot
# Политика перезапуска процесса,если он завершится с ошибкой (ненулевой код возврата)
Restart=on-failure
RestartSec=10

[Install]
# Служба будет автоматически запускаться при загрузке системы
# Доступна уже из многопользовательского режима
WantedBy=multi-user.target
```

Команда `subjob` — это не встроенная команда системы, а скрипт, установленный через `pip`. Он лежит в:
- `/home/subjob/.local/bin/subjob` (при установке с `--user`)
- `/opt/subscribe_job_rss/.venv/bin/subjob` (в виртуальном окружении)

Без указания `PATH`, `systemd` не найдёт `subjob` и не запустит службу (выдаст `command not found`).

Пользователь, используемый для запуска приложения (в данном случае `subjob`) должен существовать в системе и иметь права на чтение/запись в рабочую директорию
```bash
sudo useradd -c "Application user for jubjob.service" subjob
sudo chown -R subjob:subjob /opt/subscribe_job_rss
```

Команды управления службой:
```bash
# Создать файл сервиса (описан выше)
sudo vim /etc/systemd/system/subjob.service

# Перезапустить демона systemctl, чтобы он увидел новый юнит
sudo systemctl daemon-reload

# Запуск и проверка состояния сервиса
sudo systemctl enable --now subjob.service
sudo systemctl status subjob.service

# Вывести логи сервиса
sudo journalctl -u subjob.service -f
```
